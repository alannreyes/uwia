🎯 Executive Summary

The UWIA (Underwriting Intelligence Automation) system automates insurance document analysis using n8n as orchestrator and a NestJS API with Gemini 2.5 Pro for intelligent processing. The workflow processes multiple PDF documents from Google Drive and extracts specific information using generative AI.

📊 Production Key Metrics

  • Main Endpoint: POST /api/underwriting/evaluate-gemini
  • AI Model: Gemini 2.5 Pro (NOT Gemini 1.5 Pro)
  • File Limit: 350MB (supports giant POLICY files)
  • Dominant Method: Inline API (~95% of cases)
  • Typical Performance: 8-16 seconds per document < 20MB
  • Parallel Processing: 6 documents simultaneously per case

Processing Methods (Production Reality)

Method Real Usage Size Performance Typical Cases
1. Inline API 95% < 20MB 8-16s LOP, CERTIFICATE, WEATHER, ROOF
2. File API 4% 20-50MB 20-30s Medium POLICY files
3. Page Splitting 1% > 50MB 20s/chunk Giant POLICY (340MB, 11 chunks)

Typical Workflow

n8n → Google Drive → 6 PDFs in parallel → /evaluate-gemini →
→ Gemini Inline API (< 20MB) → Consolidation → ClaimPay

Real Case: Record 20103274 processed 6 documents (LOP, CERTIFICATE, WEATHER, ROOF, POLICY 340MB, INVOICES) in ~240 seconds total.

🔄 System Architecture

n8n Workflow - Step by Step

[Webhook] → [Validate Input] → [List PDFs] → [Filter] → [Download] → [API Gemini] → [Consolidate] → [Send Result]

1. 🚀 Webhook Trigger

Why: The webhook is the entry point that receives the processing request with all case information. It acts as the trigger that initiates the entire automation, receiving unique identifiers for the case (record_id) and the Google Drive folder (carpeta_id) where documents reside, plus optional context to improve extraction accuracy through comparisons.

  • Endpoint: POST https://n8n.claimpay.net/webhook/UWIA
  • Required fields: carpeta_id, record_id
  • Optional context: insured_name, insurance_company, date_of_loss, etc.

2. ✅ Input Validation

Why: This step normalizes input to ensure the system can process any data structure it receives. It separates critical mandatory fields (carpeta_id, record_id) from variable context, allowing flexibility in additional data. Context is grouped into a JSON object to inject it later into prompts via template variables, improving extraction accuracy without hardcoding specific fields in the workflow.

  • Validates mandatory fields: carpeta_id, record_id
  • Groups everything else into context object
  • Generates timestamp for tracking

3. 📁 Document Search

Why: Connects to Google Drive API to get all available PDFs in the case folder. The filter excludes consolidated documents (ITL_SUM) to avoid processing previous results. This step is critical because it determines which documents will be analyzed, allowing the system to automatically discover available files without manual configuration per case.

  • Searches PDFs in Google Drive using carpeta_id
  • Query: mimeType='application/pdf' and not name contains 'ITL_SUM'
  • Excludes: ITL_SUM files, folders, shortcuts

4. 🔍 Filtering and Preparation

Why: Filters files to ensure only valid PDFs are processed. Alphabetical sorting guarantees predictable and consistent processing between runs. n8n does NOT filter by file size - all PDFs are downloaded and sent to the API regardless of size. The API is responsible for enforcing the size limit (350MB), not n8n. This design allows the system to handle extremely large files (like 340MB POLICY files) without n8n intervention.

  • File type filtering: Only valid PDF files (excludes ITL_SUM)
  • Alphabetical sorting: Ensures consistent processing order
  • NO size limit in n8n: All files sent to API (API enforces 350MB limit)
  • Real case: POLICY 340MB successfully processed (proves no n8n size limit)

5. ⬇️ File Download

Why: Downloads binary content of each PDF from Google Drive and prepares it for transmission to the API. This step is necessary because the API requires the complete file in memory to perform intelligent analysis (document type detection, OCR if needed, text extraction). Binary conversion optimizes transfer in the multipart/form-data request.

6. 🤖 AI Processing

Why: Sends each document to the intelligent processing API along with its context. The API automatically determines document type (POLICY, LOP, WEATHER, etc.) and selects the optimal processing method based on PDF size and characteristics. The JSON context allows prompts to be dynamically personalized with case information, improving extraction accuracy through cross-validations.

  • Endpoint: POST http://automate_uwia_qa:5045/api/underwriting/evaluate-gemini
  • Method: multipart/form-data
  • Payload: carpeta_id, record_id, document_name, context (JSON), file (binary)

⚙️ Processing Methods

🔸 Method 1: Gemini Inline API (PRIMARY METHOD)

Why: For most documents (LOP, CERTIFICATE, WEATHER, ROOF typically < 20MB), sending the PDF encoded in base64 directly in the request is more efficient than uploading it to Google Cloud Storage. It reduces latency (eliminates upload step) and costs (doesn't use temporary storage). Google officially recommends this method for PDFs < 20MB because their infrastructure can process them in memory without performance issues.

Production Usage

  • For: Files < 20MB (MAJORITY of cases in production: ~95%)
  • Real usage: LOP (1.41MB), CERTIFICATE (0.40MB), WEATHER (0.21MB), ROOF (2.17MB)
  • Advantages: Fast response, lower cost, no temporary storage
  • Performance: ~8-16 seconds per document
  • Google official limit: 20MB (according to Gemini API 2025 documentation)
  • Code: gemini-file-api.service.ts:265-307

🔸 Method 2: Gemini File API (SECONDARY USE)

Why: Files between 20-50MB exceed the Inline API in-memory processing limit but are within the File API limit for PDFs. This method uploads the file to Google Cloud Storage temporarily, allowing Gemini servers to process it in a distributed manner with more resources. It's essential for scanned documents because it activates Google's advanced OCR (better accuracy than simple text extraction).

Production Usage

  • For: Files 20-50MB (medium cases, secondary use in production)
  • Advantages: Advanced OCR, distributed processing, large file handling
  • Google official limit: 50MB for PDFs (Google enforced PDF limit)
  • Process: Upload → Cloud Storage → Processing → Cleanup
  • Code: gemini-file-api.service.ts:184-260

🔸 Method 3: Page-Based Splitting + File API (EXTREME CASES)

Why: Files > 50MB exceed Google's official limit for PDFs in the File API. The only way to process them is to divide them into smaller fragments. Page-based division (with pdf-lib) is used because it preserves the visual integrity of each page, critical for OCR and document analysis. Each fragment (chunk) is processed with File API (method 2) and partial responses are intelligently consolidated: for boolean fields (YES/NO), if any fragment finds positive evidence it prevails; for specific data (dates, amounts), found values are prioritized over NOT_FOUND.

Extreme Cases Usage

  • For: Files > 50MB up to 350MB (extreme cases: POLICY from specific clients)
  • Real usage: POLICY 340MB → 87 pages → 11 chunks of ~8 pages
  • Dynamic division: ~8 pages per chunk (~29MB per chunk for safety under 50MB limit)
  • Processing: Each chunk uses File API (method 2)
  • Consolidation: Prioritizes positive evidence (YES > NO > NOT_FOUND)
  • Performance: ~20 seconds per chunk (340MB = ~11 chunks = ~220 seconds total)
  • Code: gemini-file-api.service.ts:504-561 (splitting), 750-929 (consolidation)
  • Real log example: 📑 [PAGE-SPLIT] Dividing 87 pages into chunks of ~8 pages (3.91MB/page)

🔄 Arquitectura del Sistema

Flujo n8n - Paso a Paso

[Webhook] → [Validar Entrada] → [Listar PDFs] → [Filtrar] → [Descargar] → [API Gemini] → [Consolidar] → [Enviar Resultado]

1. 🚀 Inicio del Flujo (Webhook)

Porqué: El webhook es el punto de entrada al sistema que recibe la solicitud de procesamiento con toda la información del caso. Actúa como trigger que desencadena toda la automatización, recibiendo los identificadores únicos del caso (record_id) y la carpeta de Google Drive (carpeta_id) donde residen los documentos, además de contexto opcional para mejorar la precisión de extracción mediante comparaciones.

  • Endpoint: POST https://n8n.claimpay.net/webhook/UWIA
  • Campos requeridos: carpeta_id, record_id
  • Contexto opcional: insured_name, insurance_company, date_of_loss, etc.

2. ✅ Validación y Procesamiento

Porqué: Este paso normaliza la entrada para garantizar que el sistema pueda procesar cualquier estructura de datos que reciba. Separa los campos críticos obligatorios (carpeta_id, record_id) del contexto variable, permitiendo flexibilidad en los datos adicionales. El contexto se agrupa en un objeto JSON para inyectarlo posteriormente en los prompts mediante template variables, mejorando la precisión de extracción sin hardcodear campos específicos en el flujo.

  • Valida campos obligatorios: carpeta_id, record_id
  • Agrupa todo lo demás en objeto context
  • Genera timestamp para tracking

3. 📁 Búsqueda de Documentos

Porqué: Conecta con Google Drive API para obtener todos los PDFs disponibles en la carpeta del caso. El filtro excluye documentos consolidados (ITL_SUM) para evitar procesar resultados previos. Este paso es crítico porque determina qué documentos se analizarán, permitiendo que el sistema descubra automáticamente los archivos disponibles sin configuración manual por caso.

  • Busca PDFs en Google Drive usando carpeta_id
  • Query: mimeType='application/pdf' and not name contains 'ITL_SUM'
  • Excluye: Archivos ITL_SUM, carpetas, shortcuts

4. 🔍 Filtrado y Preparación

Porqué: Filtra archivos para asegurar que solo PDFs válidos sean procesados. El ordenamiento alfabético garantiza procesamiento predecible y consistente entre ejecuciones. n8n NO filtra por tamaño de archivo - todos los PDFs se descargan y envían al API sin importar su tamaño. El API es responsable de aplicar el límite de tamaño (350MB), no n8n. Este diseño permite que el sistema maneje archivos extremadamente grandes (como POLICY de 340MB) sin intervención de n8n.

  • Filtrado por tipo: Solo archivos PDF válidos (excluye ITL_SUM)
  • Ordenamiento alfabético: Asegura orden de procesamiento consistente
  • SIN límite de tamaño en n8n: Todos los archivos se envían al API (API aplica límite de 350MB)
  • Caso real: POLICY de 340MB procesado exitosamente (prueba que no hay límite en n8n)

5. ⬇️ Descarga de Archivos

Porqué: Descarga el contenido binario de cada PDF desde Google Drive y lo prepara para transmisión al API. Este paso es necesario porque el API requiere el archivo completo en memoria para realizar análisis inteligente (detección de tipo de documento, OCR si es necesario, extracción de texto). La conversión a binario optimiza la transferencia en el request multipart/form-data.

6. 🤖 Procesamiento con IA

Porqué: Envía cada documento al API de procesamiento inteligente junto con su contexto. El API determina automáticamente el tipo de documento (POLICY, LOP, WEATHER, etc.) y selecciona el método de procesamiento óptimo según tamaño y características del PDF. El contexto JSON permite que los prompts se personalicen dinámicamente con información del caso, mejorando la precisión de extracción mediante validaciones cruzadas.

  • Endpoint: POST http://automate_uwia_qa:5045/api/underwriting/evaluate-gemini
  • Método: multipart/form-data
  • Payload: carpeta_id, record_id, document_name, context (JSON), file (binario)

⚙️ Métodos de Procesamiento

🔸 Método 1: Gemini Inline API (MÉTODO PRINCIPAL)

Porqué: Para la mayoría de documentos (LOP, CERTIFICATE, WEATHER, ROOF típicamente < 20MB), enviar el PDF codificado en base64 directamente en el request es más eficiente que subirlo a Google Cloud Storage. Reduce latencia (elimina paso de upload) y costos (no usa almacenamiento temporal). Google recomienda oficialmente este método para PDFs < 20MB porque su infraestructura puede procesarlos en memoria sin problemas de performance.

Uso en Producción

  • Para: Archivos < 20MB (MAYORÍA de casos en producción: ~95%)
  • Uso real: LOP (1.41MB), CERTIFICATE (0.40MB), WEATHER (0.21MB), ROOF (2.17MB)
  • Ventajas: Respuesta rápida, menor costo, sin almacenamiento temporal
  • Performance: ~8-16 segundos por documento
  • Límite oficial Google: 20MB (según documentación Gemini API 2025)
  • Código: gemini-file-api.service.ts:265-307

🔸 Método 2: Gemini File API (USO SECUNDARIO)

Porqué: Archivos entre 20-50MB exceden el límite de procesamiento en memoria del Inline API pero están dentro del límite para PDFs del File API. Este método sube el archivo a Google Cloud Storage temporalmente, permitiendo que los servidores de Gemini lo procesen de forma distribuida con más recursos. Es esencial para documentos escaneados porque activa el OCR avanzado de Google (mejor precisión que extracción textual simple).

Uso en Producción

  • Para: Archivos 20-50MB (casos medianos, uso secundario en producción)
  • Ventajas: OCR avanzado, procesamiento distribuido, manejo de archivos grandes
  • Límite oficial Google: 50MB para PDFs (Google enforced PDF limit)
  • Proceso: Upload → Cloud Storage → Processing → Cleanup
  • Código: gemini-file-api.service.ts:184-260

🔸 Método 3: Page-Based Splitting + File API (CASOS EXTREMOS)

Porqué: Archivos > 50MB exceden el límite oficial de Google para PDFs en el File API. La única forma de procesarlos es dividirlos en fragmentos más pequeños. Se usa división por páginas (con pdf-lib) porque preserva la integridad visual de cada página, crítico para OCR y análisis de documentos. Cada fragmento (chunk) se procesa con File API (método 2) y las respuestas parciales se consolidan inteligentemente: para campos booleanos (YES/NO), si algún fragmento encuentra evidencia positiva prevalece; para datos específicos (fechas, montos), se priorizan valores encontrados sobre NOT_FOUND.

Uso en Casos Extremos

  • Para: Archivos > 50MB hasta 350MB (casos extremos: POLICY de clientes específicos)
  • Uso real: POLICY de 340MB → 87 páginas → 11 chunks de ~8 páginas
  • División dinámica: ~8 páginas por chunk (~29MB por chunk para seguridad bajo límite de 50MB)
  • Procesamiento: Cada chunk usa File API (método 2)
  • Consolidación: Prioriza evidencia positiva (YES > NO > NOT_FOUND)
  • Performance: ~20 segundos por chunk (340MB = ~11 chunks = ~220 segundos total)
  • Código: gemini-file-api.service.ts:504-561 (splitting), 750-929 (consolidation)
  • Ejemplo real log: 📑 [PAGE-SPLIT] Dividiendo 87 páginas en chunks de ~8 páginas (3.91MB/página)

# ===== Rate Limiting Gemini =====
GEMINI_RATE_LIMIT_RPM=80                  # 80 requests por minuto
GEMINI_RATE_LIMIT_TPM=1500000             # 1.5M tokens por minuto
GEMINI_MAX_RETRIES=3                      # 3 reintentos en caso de fallo

Decisión de Límites

Configuración Valor Razón
MAX_FILE_SIZE 350MB Cliente solicitó soporte para archivos POLICY extremadamente grandes (hasta 340MB observados en producción). Límite configurado con margen de seguridad.
GEMINI_TIMEOUT 120000 (2 min) Suficiente para archivos < 50MB con Inline/File API. Para archivos > 50MB se usa PDF_PROCESSING_TIMEOUT=1200000 (20 min).
MEMORY_LIMIT_MB 1536 Balance entre procesamiento de archivos grandes y estabilidad del contenedor Docker. Permite manejar chunks de ~30MB en memoria sin OOM errors.
MAX_PARALLEL_CHUNKS 10 Evita saturar API de Gemini con demasiados requests simultáneos. 10 chunks en paralelo = ~10 archivos procesándose a la vez en casos extremos.

🔍 Hallazgos Clave: Teoría vs Realidad

❌ Mitos Desmentidos

  1. MITO: "File API es el método principal"
    REALIDAD: Inline API procesa ~95% de archivos en producción (< 20MB)
  2. MITO: "Límite de archivos es 30MB"
    REALIDAD: MAX_FILE_SIZE=350MB en producción para soportar POLICY gigantes
  3. MITO: "Usamos Modern RAG con embeddings"
    REALIDAD: Modern RAG NO se usa en producción. Sistema usa Page Splitting + File API
  4. MITO: "Gemini 1.5 Pro es el modelo"
    REALIDAD: GEMINI_MODEL=gemini-2.5-pro en producción
  5. MITO: "Archivos ≥20MB van a File API automáticamente"
    REALIDAD: Solo en endpoint legacy. /evaluate-gemini decide en 2 capas (50MB y 20MB)

✅ Verdades Confirmadas

  1. Inline API domina: 95% de casos (LOP 1.41MB, CERTIFICATE 0.40MB, WEATHER 0.21MB, ROOF 2.17MB)
  2. Page Splitting funciona: POLICY 340MB → 11 chunks → procesamiento exitoso
  3. Performance consistente: ~8-16s para < 20MB, ~20s por chunk para archivos grandes
  4. Consolidación inteligente: YES prevalece sobre NO cuando chunks tienen evidencia parcial
  5. Throughput real: 6 documentos en paralelo sin saturar rate limits de Gemini

🎯 Ventajas del Sistema (Validadas en Producción)

Escalabilidad

  • Procesamiento paralelo: n8n envía 6 documentos simultáneamente por caso
  • Enrutamiento inteligente: 3 métodos automáticos según tamaño (< 20MB, 20-50MB, > 50MB)
  • Límites extremos: Soporta hasta 350MB por archivo (POLICY gigantes)
  • Chunking dinámico: Divide automáticamente archivos > 50MB en fragmentos óptimos

Flexibilidad

  • Prompts DB-first: Actualizables sin redespliegue desde tabla document_consolidado
  • Variables dinámicas: Template variables (%insured_name%) inyectadas en runtime
  • Campos consolidados: Respuestas semicolon-separated mapeadas a 31+ campos individuales
  • Documentos opcionales: INVOICES sin prompt configurado → respuesta vacía graceful

Robustez Extrema

  • Detección automática: PDFs escaneados activan File API independiente del tamaño
  • Timeouts extendidos: 20 minutos máximo para archivos ultra grandes
  • Monitoreo de memoria: 1.5GB límite con alertas automáticas
  • Retry logic: 3 reintentos automáticos en Gemini API con backoff exponencial
  • Consolidación inteligente: YES prevalece sobre NO en chunks (evidencia positiva)
